## Summary

Phase 16 builds an ML-based component placement engine that predicts optimal (x, y, rotation) coordinates for PCB components given a schematic netlist. This phase extends the existing rule-based `PlacementEngine` (grid placement with clearance validation) with a learned model that incorporates netlist topology, spatial constraints, and DRC rules. The system converts schematic netlists into graph representations, trains a placement prediction model using the Phase 9 GRPO pipeline, and validates placements against DRC clearance checks using the Phase 8 spatial query engine.

The primary technical challenge is graph representation: PCB netlists are naturally hypergraphs (multi-pin nets connect more than two components), but standard GNN layers operate on pairwise edges. The research recommends a bipartite component-net graph with attention-based message passing, avoiding the need for PyTorch Geometric (not installed) by implementing a lightweight attention layer directly in PyTorch. This keeps the dependency footprint minimal while leveraging the existing PyTorch 2.10 runtime.

The secondary challenge is training data. Phase 13 (Real-World PCB Training Pipeline) is a prerequisite that produces real board graph datasets. Before Phase 13 produces data, the placement model can train on synthetic boards generated by extending the Phase 8 maze generator pattern to produce component placement samples with known-optimal positions from simulated annealing.

**Primary recommendation:** Use a custom attention-based graph neural network implemented in vanilla PyTorch over a bipartite component-net graph. Train with GRPO using the existing Phase 9 training infrastructure. Integrate with Phase 8 spatial primitives for DRC validation. Keep the rule-based PlacementEngine as a fallback and scoring baseline.

### Pattern 1: Bipartite Component-Net Graph Representation (PLACE-01)

**What:** Convert schematic netlist into a bipartite graph where one partition is components and the other is nets. Each component node connects to net nodes it participates in.

**When to use:** Every placement prediction request.

**Why bipartite over hypergraph:** PCB nets are hyperedges (connecting 2+ components). A bipartite graph naturally represents this: a net with 5 pins becomes 5 edges through a single net node, preserving the shared-net relationship without exploding the edge count. Standard GNN message passing works directly on bipartite graphs.

**Example:**

```python
# Source: [ASSUMED] - standard graph representation for VLSI placement
import networkx as nx
from kicad_agent.generation.intent import ComponentSpec, NetSpec

def netlist_to_placement_graph(
    components: list[ComponentSpec],
    nets: list[NetSpec],
    board_width: float,
    board_height: float,
) -> nx.Graph:
    """Convert netlist to bipartite placement graph.

    Node types:
      - "component": reference, library_id, value, estimated_size, position (if fixed)
      - "net": name, pin_count, is_power, criticality_weight

    Edge attributes:
      - pin_count: number of pins connecting this component to this net
    """
    G = nx.Graph()
    G.graph["board_width"] = board_width
    G.graph["board_height"] = board_height

    # Add component nodes (partition 0)
    for comp in components:
        G.add_node(
            f"comp:{comp.reference}",
            bipartite=0,
            node_type="component",
            reference=comp.reference,
            library_id=comp.library_id,
            value=comp.value,
            estimated_size=_estimate_component_size(comp),
            is_fixed=comp.position is not None,
            fixed_x=comp.position.x if comp.position else None,
            fixed_y=comp.position.y if comp.position else None,
        )

    # Add net nodes (partition 1) and edges
    for net in nets:
        # Extract unique component references from pins
        comp_refs = set()
        for pin in net.pins:
            parts = pin.split(".")
            if parts:
                comp_refs.add(parts[0])

        net_node_id = f"net:{net.name}"
        G.add_node(
            net_node_id,
            bipartite=1,
            node_type="net",
            name=net.name,
            pin_count=len(net.pins),
            is_power=net.name in ("GND", "VCC", "+3V3", "+5V", "VDD"),
            criticality=3.0 if net.name in ("GND", "VCC") else 1.0,
        )

        # Connect each participating component to the net node
        for ref in comp_refs:
            comp_node = f"comp:{ref}"
            if G.has_node(comp_node):
                G.add_edge(comp_node, net_node_id, weight=1.0)

    return G
```

### Pattern 2: Attention-Based Placement Model (PLACE-02)

**What:** A custom graph neural network using multi-head attention for message passing between component and net nodes, with per-component output heads predicting (x, y, rotation).

**When to use:** Every placement prediction after graph construction.

**Why custom instead of PyG:** PyTorch Geometric is not installed and adding it introduces a heavy dependency (requires matching CUDA/torch version). A 2-3 layer attention network is ~200 lines and provides full control over the message passing scheme.

```python
# Source: [ASSUMED] - attention mechanism adapted for bipartite PCB placement graphs
import torch
import torch.nn as nn

class BipartiteAttentionLayer(nn.Module):
    """Multi-head attention for bipartite component-net graphs.

    Message passing: component nodes attend to their connected net nodes,
    aggregate weighted features, then net nodes attend back to components.
    """
    def __init__(self, comp_dim: int, net_dim: int, n_heads: int = 4):
        super().__init__()
        self.n_heads = n_heads
        self.comp_proj = nn.Linear(comp_dim, comp_dim)
        self.net_proj = nn.Linear(net_dim, comp_dim)  # project net to comp space
        self.attention = nn.MultiheadAttention(comp_dim, n_heads, batch_first=True)

    def forward(self, comp_features, net_features, adj_matrix):
        """
        Args:
            comp_features: (batch, n_comp, comp_dim)
            net_features: (batch, n_net, net_dim)
            adj_matrix: (batch, n_comp, n_net) binary adjacency
        Returns:
            updated comp_features: (batch, n_comp, comp_dim)
        """
        projected_nets = self.net_proj(net_features)
        # Attention: components query against net keys/values
        Q = self.comp_proj(comp_features)
        K = projected_nets
        V = projected_nets
        attn_out, _ = self.attention(Q, K, V)
        return comp_features + attn_out  # residual


class PlacementModel(nn.Module):
    """Predicts (x, y, rotation) for each component from netlist graph.

    Architecture:
      1. Node feature embedding (component type, size + net criticality)
      2. 3 rounds of bipartite attention message passing
      3. Per-component MLP heads: x, y, rotation
    """
    def __init__(
        self,
        comp_feature_dim: int = 32,
        net_feature_dim: int = 16,
        hidden_dim: int = 128,
        n_layers: int = 3,
        n_heads: int = 4,
    ):
        super().__init__()
        self.comp_embed = nn.Linear(comp_feature_dim, hidden_dim)
        self.net_embed = nn.Linear(net_feature_dim, hidden_dim)

        self.attn_layers = nn.ModuleList([
            BipartiteAttentionLayer(hidden_dim, hidden_dim, n_heads)
            for _ in range(n_layers)
        ])
        self.layer_norms = nn.ModuleList([
            nn.LayerNorm(hidden_dim) for _ in range(n_layers)
        ])

        # Output heads
        self.x_head = nn.Linear(hidden_dim, 1)
        self.y_head = nn.Linear(hidden_dim, 1)
        self.rot_head = nn.Linear(hidden_dim, 1)

    def forward(self, comp_features, net_features, adj_matrix, board_w, board_h):
        """
        Args:
            comp_features: (batch, n_comp, comp_feature_dim)
            net_features: (batch, n_net, net_feature_dim)
            adj_matrix: (batch, n_comp, n_net)
            board_w: (batch,) board width
            board_h: (batch,) board height
        Returns:
            predictions: (batch, n_comp, 3) -- (x, y, rotation_degrees)
        """
        h_comp = torch.relu(self.comp_embed(comp_features))
        h_net = torch.relu(self.net_embed(net_features))

        for attn, ln in zip(self.attn_layers, self.layer_norms):
            h_comp = ln(attn(h_comp, h_net, adj_matrix))
            h_comp = torch.relu(h_comp)

        x = torch.sigmoid(self.x_head(h_comp)).squeeze(-1) * board_w.unsqueeze(-1)
        y = torch.sigmoid(self.y_head(h_comp)).squeeze(-1) * board_h.unsqueeze(-1)
        rot = (torch.sigmoid(self.rot_head(h_comp)).squeeze(-1) * 360.0) - 180.0

        return torch.stack([x, y, rot], dim=-1)
```

### Pattern 3: DRC-Aware Placement Scoring (PLACE-03, PLACE-04)

**What:** Combine existing spatial validation with ML-optimized quality metrics.

**When to use:** After every placement prediction, during training loss computation, and for final placement scoring.

```python
# Source: [VERIFIED: codebase] - extends existing validate_placement_clearance()
# and PlacementEngine.score_placement()

from kicad_agent.generation.placement import validate_placement_clearance
from kicad_agent.spatial.primitives import SpatialBox

def compute_placement_loss(
    predictions: dict[str, tuple[float, float, float]],  # ref -> (x, y, rot)
    graph,  # PlacementGraph
    board_width: float,
    board_height: float,
    min_clearance: float = 1.0,
) -> dict[str, float]:
    """Compute differentiable placement quality metrics.

    Returns dict of loss components for GRPO reward computation.
    """
    # 1. Wirelength (HPWL - Half-Perimeter Wirelength)
    hpwl = _compute_hpwl(predictions, graph)

    # 2. Clearance violations (non-differentiable, used as penalty weight)
    boxes = _predictions_to_boxes(predictions)
    clearance_result = validate_placement_clearance(
        boxes,
        min_clearance_mm=min_clearance,
        board_bounds=(min_clearance, min_clearance,
                      board_width - min_clearance, board_height - min_clearance),
    )
    violation_count = len(clearance_result["violations"])

    # 3. Overlap penalty
    overlap_area = _compute_overlap_area(predictions, min_clearance)

    # 4. Edge proximity penalty
    edge_penalty = _compute_edge_penalty(predictions, board_width, board_height, min_clearance)

    return {
        "hpwl": hpwl,
        "clearance_violations": float(violation_count),
        "overlap_area": overlap_area,
        "edge_penalty": edge_penalty,
        "total_loss": hpwl + 10.0 * violation_count + 5.0 * overlap_area + edge_penalty,
    }
```

### Pattern 4: Interactive Constraint Propagation (PLACE-05)

**What:** User places some components, AI places the rest respecting those fixed positions as hard constraints.

**When to use:** Interactive placement mode where the user has partially populated a board.

```python
# Source: [ASSUMED] - constrained optimization pattern for partial placement

from scipy.optimize import dual_annealing
import numpy as np

def interactive_placement(
    graph,
    fixed_positions: dict[str, tuple[float, float, float]],  # user-placed
    board_width: float,
    board_height: float,
    model_predictions: dict[str, tuple[float, float, float]] | None = None,
) -> dict[str, tuple[float, float, float]]:
    """Place remaining components respecting user-fixed positions.

    1. Use model predictions as initial guess (if available)
    2. Run simulated annealing with fixed positions as constraints
    3. Validate against DRC rules
    """
    # Identify free components (not in fixed_positions)
    all_refs = [n for n, d in graph.nodes(data=True) if d.get("node_type") == "component"]
    free_refs = [r for r in all_refs if r not in fixed_positions]

    if not free_refs:
        return fixed_positions

    # Initial positions: model predictions or grid fallback
    x0 = np.array([
        model_predictions.get(ref, (board_width/2, board_height/2, 0.0))[:2]
        for ref in free_refs
    ]).flatten()

    # Objective: minimize HPWL + clearance penalty
    def objective(params):
        positions = {}
        positions.update(fixed_positions)
        for i, ref in enumerate(free_refs):
            x = np.clip(params[2*i], 1.0, board_width - 1.0)
            y = np.clip(params[2*i+1], 1.0, board_height - 1.0)
            positions[ref] = (x, y, 0.0)
        return _compute_hpwl(positions, graph) + 10.0 * _count_violations(positions, min_clearance=1.0)

    bounds = [(1.0, board_width - 1.0), (1.0, board_height - 1.0)] * len(free_refs)
    result = dual_annealing(objective, bounds, x0=x0, maxiter=500, seed=42)

    # Build final positions
    final = dict(fixed_positions)
    for i, ref in enumerate(free_refs):
        final[ref] = (result.x[2*i], result.x[2*i+1], 0.0)

    return final
```

### Anti-Patterns to Avoid

- **Pairwise component edges (not bipartite):** Creating an edge between every pair of components sharing a net causes O(n^2) edge explosion for power nets. Use bipartite component-net representation instead. [VERIFIED: codebase - existing PlacementEngine.score_placement() has this issue in wire_length computation]
- **Predicting absolute coordinates directly without normalization:** Raw (x, y) predictions are unbounded and hard to train. Use sigmoid output scaled to board dimensions (see PlacementModel.forward()). [ASSUMED]
- **Ignoring component rotation:** Many footprints (e.g., SOIC, QFP) have significantly different bounding boxes at different rotations. The model must predict rotation as a third output. [ASSUMED]
- **Training only on synthetic maze data:** Maze routing data teaches spatial reasoning but not placement topology. Real-world placement data from Phase 13 is essential for practical quality. [ASSUMED]
- **Replacing the existing PlacementEngine entirely:** The rule-based engine provides deterministic fallback, baseline scoring, and is already tested (13 tests). The ML model should enhance, not replace. [VERIFIED: codebase - placement.py has 13 tests]

## Common Pitfalls

### Pitfall 1: Power Net Edge Explosion

**What goes wrong:** GND and VCC nets connect to nearly every component. In a pairwise graph, this creates O(n^2) edges for power nets alone, dominating the graph structure and causing the model to overweight power net proximity.
**Why it happens:** Power nets are fundamentally different from signal nets -- they have high fanout but low routing complexity (typically handled with copper pours, not individual traces).
**How to avoid:** In the bipartite representation, assign lower criticality weights to power nets (criticality=1.0 vs 3.0 for high-speed signals). Consider excluding power nets from the graph entirely and handling them with the existing decoupling cap placement logic.
**Warning signs:** Model predicts all components clustered near board center (overfitting to power net connectivity).

### Pitfall 2: Component Size Ignorance

**What goes wrong:** The model places a large QFP IC and a tiny 0402 resistor in the same spot because it only learned from connectivity, not geometry.
**Why it happens:** Graph neural networks operate on abstract node features. If component size is not encoded as a node feature, the model has no spatial awareness of footprint dimensions.
**How to avoid:** Include estimated component size as a node feature (reuse PlacementEngine._estimate_size() heuristic: IC=10, FET=8, inductor=5, R/C=2). For production, use actual footprint bounding box dimensions from the library.
**Warning signs:** Clearance validation reports many violations after model prediction.

### Pitfall 3: Training Data Distribution Mismatch

**What goes wrong:** Model trained on synthetic 10-component boards performs poorly on real 50-component boards with heterogeneous component types.
**Why it happens:** Synthetic data from the maze generator produces uniform component distributions that don't reflect real design patterns (IC clusters, decoupling cap proximity, connector placement at board edges).
**How to avoid:** Use Phase 13 real-world data for fine-tuning. Start with synthetic data for model architecture validation, then transfer-learn on real boards. Include component count as a conditioning variable.
**Warning signs:** Model quality scores are high on synthetic test sets but low on real board benchmarks.

### Pitfall 4: Non-Differentiable DRC Loss

**What goes wrong:** Trying to backpropagate through DRC violation counts (integer, non-differentiable) causes NaN gradients or training collapse.
**Why it happens:** Clearance violations are binary (violated/not violated). Count-based loss has zero gradient almost everywhere.
**How to avoid:** Use continuous proxy losses for training: sigmoid-smoothed distance penalty instead of binary violation count. Reserve exact DRC checks for evaluation, not training. The existing validate_placement_clearance() is for inference-time validation only.
**Warning signs:** Training loss plateaus at a high value, or gradients become NaN after a few steps.

### Pitfall 5: Fixed Component Constraint Violation in Interactive Mode

**What goes wrong:** The SA refinement or model prediction moves user-placed components to "better" positions, violating the explicit user constraint.
**Why it happens:** The optimizer doesn't distinguish between free and fixed components in its objective function.
**How to avoid:** Partition components into fixed/free sets before optimization. Fixed components never appear in the SA parameter vector. In the GNN, mask fixed component outputs so the loss doesn't try to move them.
**Warning signs:** User reports that their manually placed components "shifted" after AI placement.

### Netlist-to-Graph Feature Extraction

```python
# Source: [VERIFIED: codebase] - uses existing ComponentSpec, NetSpec schemas
import numpy as np

def extract_component_features(
    comp_spec,  # ComponentSpec from generation.intent
    board_width: float,
    board_height: float,
) -> np.ndarray:
    """Extract fixed-size feature vector for a component node.

    Features (32-dim):
      [0]: estimated_size (from PlacementEngine._estimate_size)
      [1]: is_ic (1 if reference starts with 'U')
      [2]: is_passive (1 if R or C)
      [3]: is_connector (1 if J)
      [4]: is_fixed (1 if position is specified)
      [5]: normalized_fixed_x (x / board_width, 0 if not fixed)
      [6]: normalized_fixed_y (y / board_height, 0 if not fixed)
      [7-15]: library_id character hash (8-dim embedding)
      [16-23]: value string hash (8-dim embedding)
      [24-31]: zero-padded reserved
    """
    ref = comp_spec.reference.upper()
    features = np.zeros(32, dtype=np.float32)

    # Heuristic size (same as PlacementEngine._estimate_size)
    features[0] = 10.0 if ref.startswith("U") else (
        8.0 if ref.startswith("Q") else (
            5.0 if ref.startswith(("L", "D")) else (
                2.0 if ref.startswith(("R", "C")) else 3.0
            )
        )
    )
    features[1] = 1.0 if ref.startswith("U") else 0.0
    features[2] = 1.0 if ref.startswith(("R", "C")) else 0.0
    features[3] = 1.0 if ref.startswith("J") else 0.0
    features[4] = 1.0 if comp_spec.position is not None else 0.0

    if comp_spec.position is not None:
        features[5] = comp_spec.position.x / board_width
        features[6] = comp_spec.position.y / board_height

    # Simple string hashing for categorical features
    for i, ch in enumerate(comp_spec.library_id[:8]):
        features[7 + i] = ord(ch) / 255.0
    for i, ch in enumerate(comp_spec.value[:8]):
        features[16 + i] = ord(ch) / 255.0

    return features
```

### GRPO Reward Function for Placement

```python
# Source: [VERIFIED: codebase] - extends Phase 9 reward patterns
from kicad_agent.training.grpo import GRPOTrainer

def placement_reward(
    predicted_positions: dict[str, tuple[float, float, float]],
    reference_positions: dict[str, tuple[float, float, float]],
    graph,  # PlacementGraph
    board_width: float,
    board_height: float,
) -> float:
    """Compute reward for a placement prediction.

    Higher reward = better placement. Combines:
    - Position accuracy vs reference (if available)
    - Wirelength quality (HPWL)
    - DRC clearance score
    - Edge proximity score

    Returns reward in [0, 1].
    """
    # 1. Position accuracy (only if reference available)
    if reference_positions:
        pos_errors = []
        for ref in predicted_positions:
            if ref in reference_positions:
                px, py, _ = predicted_positions[ref]
                rx, ry, _ = reference_positions[ref]
                pos_errors.append(((px - rx)**2 + (py - ry)**2)**0.5)
        avg_error = sum(pos_errors) / len(pos_errors) if pos_errors else 0.0
        board_diag = (board_width**2 + board_height**2)**0.5
        accuracy = max(0.0, 1.0 - avg_error / board_diag)
    else:
        accuracy = 0.5  # neutral when no reference

    # 2. Wirelength (reuse existing score_placement pattern)
    hpwl = _compute_hpwl(predicted_positions, graph)
    max_wire = board_diag * len(predicted_positions)
    wire_score = max(0.0, 1.0 - hpwl / max(0.001, max_wire))

    # 3. Clearance (reuse existing validation)
    boxes = _predictions_to_boxes(predicted_positions)
    result = validate_placement_clearance(boxes, min_clearance_mm=1.0)
    clearance_score = 1.0 if result["valid"] else max(0.0, 1.0 - 0.1 * len(result["violations"]))

    return 0.3 * accuracy + 0.4 * wire_score + 0.3 * clearance_score
```

### Integration with Existing PlacementEngine

```python
# Source: [VERIFIED: codebase] - wraps existing PlacementEngine for fallback
from kicad_agent.generation.placement import PlacementEngine, PlacementResult

class HybridPlacementEngine:
    """Combines ML prediction with rule-based fallback.

    Strategy:
    1. If ML model is available and trained: use prediction + SA refinement
    2. If ML model fails or produces invalid placement: fall back to grid placement
    3. Always validate with DRC regardless of source
    """
    def __init__(self, board_width: float, board_height: float, min_clearance: float = 1.0):
        self.rule_engine = PlacementEngine(board_width, board_height, min_clearance)
        self.ml_predictor = None  # PlacementPredictor loaded on demand

    def place_components(self, components, nets=None, fixed_positions=None):
        # Try ML placement first
        if self.ml_predictor is not None and self.ml_predictor.is_ready:
            try:
                result = self.ml_predictor.predict(components, nets, fixed_positions)
                if result.valid:
                    return result
            except Exception:
                pass  # Fall through to rule-based

        # Fallback to rule-based grid placement
        return self.rule_engine.place_components(components)
```

## Open Questions

1. **Phase 13 data format alignment**
   - What we know: Phase 13 produces real-world PCB graph datasets. Phase 16 needs component-level placement training data.
   - What's unclear: Whether Phase 13's graph format includes component positions as training targets, or only connectivity.
   - Recommendation: Define PlacementSample schema in Phase 16 that Phase 13 can populate. Coordinate during Phase 13 planning.

2. **Component footprint bounding box access**
   - What we know: Current code uses 2mm x 2mm default bounding boxes (PlacementEngine._build_boxes). Real footprints vary from 0.5mm (0402) to 30mm+ (BGA).
   - What's unclear: Whether footprint library data (.kicad_mod) is accessible at placement time for accurate bounding boxes.
   - Recommendation: Start with heuristic sizes, add footprint lookup as an enhancement. The _estimate_size() method provides 4 tiers which is sufficient for initial training.

3. **Training compute requirements**
   - What we know: PyTorch 2.10 is available. The reward model (256-dim transformer, 4 layers) trains in minutes on CPU for maze data.
   - What's unclear: Whether placement model training (larger graphs, continuous output) requires GPU for practical training times.
   - Recommendation: Start with CPU training on small boards (10-20 components). Profile and add MPS/CUDA support if training is too slow.
